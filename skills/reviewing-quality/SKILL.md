# Quality Review

Ensure every deliverable meets quality standards before it ships. Covers content, code, agent outputs, and business materials.

## When to Use

- Reviewing content before publishing (social posts, emails, copy)
- Reviewing code before deployment
- Reviewing agent outputs before they reach clients
- Reviewing business materials (proposals, reports, decks)

## Process

1. Receive the deliverable, its type, author, intended audience, and context.
2. Review across five dimensions:
   - **Factual Accuracy**: Are claims verifiable? Any fabricated details? Sources cited where needed?
   - **Brand Voice**: Sounds authentic? Banned words present? (leverage, synergy, disrupt, paradigm, revolutionary, game-changing)
   - **Technical Accuracy** (code): Works as intended? Security vulnerabilities? Edge cases handled?
   - **Bias and Safety**: Unintentional bias? Misleading statements? Promises that cannot be kept?
   - **Formatting**: Platform-appropriate? Spelling/grammar? Clear structure?
3. Classify findings by severity:
   - **BLOCK**: Must fix before publishing/deploying (factual errors, security issues, brand damage).
   - **MAJOR**: Should fix (significant quality issues).
   - **MINOR**: Nice to fix (polish items).
   - **NOTE**: Observation (no action required).
4. Identify at least one strength.
5. Issue verdict: PASS / PASS WITH CHANGES / FAIL.

## Output Format

- Verdict with rationale
- Findings table: severity, description, fix recommendation
- Strengths section
- Clear next-step recommendation

## Constraints

- Never approve something with unresolved doubts -- flag the doubt.
- Never rewrite the deliverable -- provide specific fix instructions.
- Never block for stylistic preferences -- only for substantive issues.
- Be specific: "This claim is unverified" not "Some claims may need checking."
